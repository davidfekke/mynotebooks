3

Think Like a Consultant

~20 minutes

Before you touch any code or data, good consultants take time to understand the problem. In the real world, you never jump straight into the data — you think it through first. Open your shared Google Doc and take a first pass at these questions as a team:

Understanding the Problem

  • What is the client actually trying to solve? In your own words, what's the core problem?
  • Who are the end users of these models? What do they care about most?
  • What would success look like for each of the 5 models? How would the client measure impact?

Expectations Before Seeing the Data

  • What do you expect the data to look like? What columns, what formats, what messiness?
  • What challenges do you anticipate? (missing values, class imbalance, noisy text, image quality)
  • What would surprise you? What would be a red flag?

Planning Your Approach

  • What's the first thing you'd want to do when you get the data?
  • What skills does each model require? (traditional ML, neural networks, computer vision, NLP)
  • What do you think will be the hardest model to build? The easiest? Why?
  • What tools and libraries do you think you'll need?
Why this matters: Writing down your expectations before you see the data is a powerful habit. When you finally open the data tomorrow, you'll immediately notice what matches your expectations and what doesn't — and that's where the real insights are. This is how experienced data scientists and consultants work.

Your First Branch — Edit an Existing File

~15 minutes

Everyone does this individually, at the same time. You're each going to create your own branch, edit the same file, and push it.

Step 1: Create Your Branch

# Make sure you're on main first
git checkout main

# Create a new branch with YOUR name
git checkout -b setup/yourname-intro

# Example:
git checkout -b setup/maria-intro
Note: We're using the setup/ prefix for this kickoff exercise. When working on the actual project, you'll use prefixes like feature/, fix/, data/, etc. as described in the Git Team Guide.

Step 2: Edit the README

Open README.md in your code editor. Find the Team Roles section at the bottom. Add your name next to a role.

The README has a section that looks like this:

## Team Members

| Role | Name |
|------|------|
| Data Engineering Lead | (your name here) |
| ML / DNN Lead | (your name here) |
| CNN Lead | (your name here) |
| NLP Lead | (your name here) |

Each person picks a different row and replaces (your name here) with their actual name. (These aren't final role assignments — just for this exercise.)

Step 3: Commit and Push

# Check what changed
git status
modified:   README.md

# Stage the file
git add README.md

# Commit with a clear message
git commit -m "Add Maria as CNN Lead in team roster"

# Push YOUR branch to GitHub
git push origin setup/maria-intro
  • Everyone has pushed their branch
  • Go to GitHub in your browser — you should see 4 new branches

Merge Time — Pull Requests

~25 minutes

Now comes the team part. You'll merge each person's branch into main, one at a time. Do this together as a group — share one screen.

Person 1 Goes First (The Easy One)

  1. Go to your repo on GitHub
  2. You'll see a banner: "setup/person1-intro had recent pushes" → Click "Compare & pull request"
  3. Title: Add [Name] to team roster
  4. Click "Create pull request"
  5. Have a teammate look at the changes ("Files changed" tab)
  6. Teammate clicks "Approve" (or just gives a thumbs up in chat)
  7. Click "Merge pull request" → "Confirm merge"
  8. Click "Delete branch" (keeps things clean)
This first merge will go smoothly because nobody else has changed main yet. Enjoy it — the next ones get more interesting.

Person 2 Goes Next (The Merge Conflict)

Person 2 also edited README.md — and since everyone edited the same section of the same file, you will get a merge conflict. This is intentional — you'll all practice resolving conflicts in a low-stakes setting before it matters on real code.

When you get a merge conflict, here's what to do:

  1. GitHub will show you: "This branch has conflicts that must be resolved"
  2. Click "Resolve conflicts"
  3. You'll see the conflict markers (<<<<<<< and >>>>>>>)
  4. As a team, decide what the file should look like (keep both names!)
  5. Delete the conflict markers, keep the correct content
  6. Click "Mark as resolved" → "Commit merge"
  7. Now you can merge the PR
This is the whole point of this exercise. Merge conflicts are normal and not scary. You just resolved your first one as a team. In the real project, this is exactly how you'll handle it when two people edit the same file.

Everyone Else Merges

All remaining team members create their PRs at the same time — don't wait for each other. Since everyone edited the same section of README.md, expect merge conflicts on every PR after the first one. That's the whole point — by the time you're done, every person on the team will have seen and resolved a merge conflict. Work through them together as a team. This is real teamwork — everyone helping everyone.

  • All 4 branches are merged into main
  • All 4 branches are deleted on GitHub
  • README.md on main has all 4 team member names
  • notebooks/ folder has 4 intro files
8

Sync Up — Everyone Pulls Main

~3 minutes

Everyone does this individually. Get the merged changes onto your local machine.

# Switch back to main
git checkout main

# Pull all the merged changes
git pull origin main

# Verify: you should see everyone's changes
cat README.md        # All 4 names in the table
ls notebooks/         # All 4 intro files

# Clean up your old local branch
git branch -d setup/yourname-intro
  • Everyone's local main has all 4 names and all 4 intro files
  • Everyone has deleted their old local branch
You just completed your first team Git workflow. You created branches, made changes, pushed, opened PRs, handled merge conflicts, merged, and synced. This is exactly how you'll work for the next 7 weeks.

This Isn't Like the Individual Capstone

When you explore the repo, you'll notice the notebooks/ folder is empty. Unlike the individual capstone, there's no pre-built starter code walking you through each step. You're given the folder structure, some testing scripts, output templates, and requirements — but how you build it is up to you. This is intentional. It's a lot more open-ended, and that's what makes it closer to real-world work. Your team decides the approach, writes the code from scratch, and owns every decision.

A Real Talk About Time

This capstone will require 5–15 hours per week outside of class from each team member. That's the reality of building 5 AI models, a web app, and a presentation in 7 weeks. One weekly check-in is a good start, but as the project ramps up you'll likely need more. These 7 weeks go fast. The teams that succeed are the ones that communicate constantly, use their time together wisely, and don't wait until the last minute. Stay active in your Slack channel, ask for help early, and treat every hour like it counts — because it does.
Accountability: Abishek will have periodic checkpoints throughout the project to review where each team stands. Your progress will be seen — so stay on track, keep your repo updated, and don't let things pile up at the end.

The #1 Rule Going Forward

Never push to main. Always use a branch. Always pull before you push. When in doubt, check the Git Team Guide or ask Abishek.
