Git Team Collaboration Guide

Working Together Without Breaking Everything

Final Capstone Project — AI & ML Bootcamp

The Golden Rule of Team Git Never push directly to main. Always work on a branch. Always pull before you push.

Key Git Definitions

Before diving in, here are the terms you'll see throughout this guide and use every day:

Term What It Means Analogy
Repository (Repo) Your entire project folder, tracked by Git. It contains all your code, history, and branches. Like a shared Google Drive folder that remembers every version of every file
Clone Download a copy of a remote repo to your computer so you can work on it locally. Like downloading a shared folder to your laptop for the first time
Branch A separate copy of the code where you can make changes without affecting anyone else's work. Like making a duplicate of a Google Doc to edit your section, then merging it back
Commit A snapshot of your changes with a message describing what you did. Think of it as a save point. Like hitting "Save" in a video game — you can always go back to this point
Stage (git add) Select which changed files you want to include in your next commit. Like putting items in your shopping cart before checking out
Push Upload your local commits to GitHub so your teammates can see them. Like syncing your local files up to the cloud
Pull Download the latest changes from GitHub to your local machine. Like refreshing a shared folder to get everyone else's updates
Pull Request (PR) A request on GitHub to merge your branch into main. Your teammate reviews it before it gets merged. Like submitting your draft for peer review before it goes into the final document
Merge Combine changes from one branch into another (usually your feature branch into main). Like taking your edited section and folding it back into the master document
Merge Conflict When two people edit the same lines of the same file and Git doesn't know which to keep. Like two people editing the same paragraph at the same time — someone has to decide what stays
Main The primary branch — the "official" version of your project. Should always have working code. The final, published version of your document that everyone reads
Remote (Origin) The version of your repo that lives on GitHub. origin is the default name for it. The cloud copy of your project that everyone shares

1. How Team Git Works

In your individual capstone, you probably worked directly on main. That's fine when you're solo. But with 4 people editing the same codebase, you'll overwrite each other's work instantly.

The solution: branches. Each person works on their own copy of the code (a branch), then merges it back when it's ready and reviewed.

The Flow

# Your daily workflow:
main (protected) ──────────────────────────────────────▶
  │                                    ▲
  ├── feature/model1-xgboost ─────────┤  (merge via Pull Request)
  │                                    │
  ├── feature/model3-cnn ─────────────┤
  │                                    │
  └── feature/webapp-dashboard ────────┘

Think of it like Google Docs: Instead of everyone typing in the same document at once, each person makes a copy, works on their section, then merges it back into the master document when done.

2. Setting Up Your Team Repo

First Time Only (Everyone)

# Clone the team repo
git clone https://github.com/your-org/your-team-repo.git
cd your-team-repo

# Check you're on main
git branch
* main

# Install dependencies
pip install -r requirements.txt

Protect Main Branch (Team Lead Does This Once)

On GitHub: Settings → Branches → Add rule for main

  • Check "Require a pull request before merging"
  • Check "Require approvals" (set to 1)
  • This prevents anyone from pushing directly to main

3. Your Daily Workflow

  1. Start your day: Update your local main 
    git checkout main
git pull origin main
  2. Create a branch for what you're working on 
    # Name it clearly: feature/what-you're-doing
git checkout -b feature/model1-add-xgboost
  3. Do your work, commit often 
    # Make changes to files...

# Stage specific files (NOT git add .)
git add models/model1_traditional_ml/train.py
git add models/model1_traditional_ml/predict.py

# Commit with a clear message
git commit -m "Add XGBoost model with SMOTE for class imbalance"
  4. Push your branch to GitHub 
    git push origin feature/model1-add-xgboost
  5. Open a Pull Request on GitHub

    Go to your repo on GitHub → you'll see a "Compare & pull request" button. Click it, write a short description, and assign a teammate to review.

  6. Teammate reviews → Approve → Merge

    After approval, click "Merge pull request" on GitHub. Then delete the branch.

  7. Everyone pulls the updated main 
    git checkout main
git pull origin main

5. Merge Conflicts (Don't Panic)

Merge conflicts happen when two people edit the same lines of the same file. Git doesn't know which version to keep, so it asks you to choose.

What a Conflict Looks Like

# Git shows you both versions:
<<<<<<< HEAD
model = RandomForestClassifier(n_estimators=200)
=======
model = XGBClassifier(n_estimators=300, max_depth=6)
>>>>>>> feature/model1-xgboost

How to Resolve

  1. Talk to your teammate — figure out which version is correct (or combine them)
  2. Edit the file to keep the right code (delete the <<<<, ====, and >>>> markers)
  3. Stage and commit the resolved file: 
    git add the-conflicted-file.py
git commit -m "Resolve merge conflict: keep XGBoost implementation"
Tip: Prevent conflicts by staying in your lane. If each person works in their own model folder (models/model1_traditional_ml/, models/model3_cnn/, etc.), you'll rarely have conflicts. The shared files (pipelines/data_pipeline.py, webapp/app.py) are where conflicts happen — communicate when editing these.

6. Undoing Mistakes (Your Safety Net)

Situation Command What It Does
Undo changes to a file (not committed yet) git checkout -- filename.py Restores the file to last commit
Unstage a file (added but not committed) git reset HEAD filename.py Removes from staging, keeps changes
Undo the last commit (keep changes) git reset --soft HEAD~1 Undoes commit, keeps files staged
See what changed in a file git diff filename.py Shows line-by-line differences
See all recent commits git log --oneline -10 Shows last 10 commits, one line each
Go back to a specific commit (to look at it) git checkout abc123 Detached HEAD — just looking, not editing
Get back to your branch after looking git checkout your-branch-name Returns to normal
Never run these on a team repo:
  • git push --force — overwrites everyone's work on the remote
  • git reset --hard on main — destroys commit history
  • git rebase main on a shared branch — rewrites history others depend on
If you're tempted to use any of these, ask your team or your instructor first.

7. Writing Good Commit Messages

Good Commits

  • Add SMOTE oversampling to handle class imbalance
  • Fix image resize bug causing CNN crash on test set
  • Add confusion matrix to Model 1 evaluation
  • Update data pipeline to handle missing weight column

Bad Commits

  • update
  • fix stuff
  • asdfjkl
  • final version FINAL v2
Tip: Commit early and often. Small commits that each do one thing are much easier to understand and undo than one giant commit that changes 15 files.

8. Things That Will Ruin Your Week

Mistake Why It's Bad What to Do Instead
Committing data files or model weights Bloats the repo, GitHub may reject pushes over 100MB Use .gitignore — it's already set up for you
Everyone working on main Constant conflicts, overwritten work, tears Always use feature branches
Not pulling before starting work Your branch diverges, merge becomes painful First thing every session: git pull origin main
Using git add . or git add -A Accidentally stages data files, secrets, junk Stage specific files: git add path/to/file.py
Not communicating about shared files Two people edit app.py at once → conflict Slack your team: "I'm editing app.py right now"
Hardcoded paths like /Users/john/Desktop/data.csv Works on your machine, breaks on everyone else's Use relative paths: data/raw/data.csv

Your Safety Net: .gitignore

Your repo includes a file called .gitignore. This tells Git which files to never track — even if they're sitting right there in your project folder. Git will pretend they don't exist.

Why It Matters

Without .gitignore, it's easy to accidentally commit things that don't belong in a Git repo:

  • Large data files — CSVs, images, and datasets that can be hundreds of MB. GitHub rejects files over 100MB, and once a large file is in your Git history, it's painful to remove.
  • Model weights — Trained model files (.h5, .pkl, .pt) can be enormous. These should be stored separately, not in Git.
  • API keys and secrets — .env files with passwords or API keys should never be committed. If you push a secret to GitHub, consider it compromised.
  • System junk — .DS_Store, __pycache__/, .ipynb_checkpoints/, and other auto-generated files that just clutter the repo.

What's Already Configured

The starter repo's .gitignore is already set up to exclude:

  • data/ — All raw and processed data files
  • Model weight files (*.h5, *.pkl, *.pt, *.joblib)
  • .env — Environment variables and secrets
  • Python cache files and Jupyter checkpoints
  • OS-specific files like .DS_Store
Don't delete or modify .gitignore unless you know what you're doing. It's protecting you and your teammates from a lot of potential headaches.

What If You Accidentally Commit a Large File?

If you commit a large file before .gitignore catches it (e.g., you added it with git add explicitly), here's how to fix it before you push:

# Undo the last commit but keep your changes
git reset --soft HEAD~1

# Remove the large file from staging
git reset HEAD path/to/large-file.csv

# Make sure .gitignore covers it, then re-commit without it
git add your-other-files.py
git commit -m "Your commit message"

If you already pushed a large file, don't try to fix it yourself with force commands — ask Abishek for help. It's fixable, but the steps are more involved and easy to mess up.

9. Team Tips That Actually Matter

  • Decide who owns which folders. If Maria owns models/model3_cnn/, nobody else edits files in there without talking to her first.
  • Merge to main at least twice a week. Long-lived branches that go weeks without merging become nightmare conflicts.
  • Review each other's PRs within 24 hours. Don't let PRs sit — it blocks your teammate.
  • Use Slack to coordinate. A quick "I'm about to edit the data pipeline" saves hours of conflict resolution.
  • If something goes wrong, don't panic. Git keeps history of everything. Ask for help before trying to fix it yourself with force commands.
  • Test before you push. Run your model's bulk test script before pushing. Don't break main for your teammates.
