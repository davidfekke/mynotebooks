Capstone Portfolio Guide

Take your project from classroom assignment to portfolio piece

You built a machine learning application from scratch — dataset selection, data cleaning, feature engineering, models, and a deployed web app. That's real work worth showing off. This guide walks you through making your capstone portfolio-ready: getting it on your personal GitHub, writing a README that makes people want to click, and keeping your app live for anyone to try.
1

Get It on Your Personal GitHub

Right now your capstone lives under the fsa-aiml-2511-classroom organization. That's fine for school, but for your portfolio you want it on your personal GitHub profile where recruiters and hiring managers will actually see it.

Most of you probably already have a personal copy of your repo. If so, you're good — skip ahead to Section 2. If not, here's how to set it up:

Option A: You Already Have a Personal Repo

If you've already been pushing to your own GitHub account, just make sure your latest code is pushed and you're all set. Move on to updating your README.

Option B: Create a New Personal Repo

# 1. Go to github.com and create a new empty repository # Name it something clean like: housing-price-predictor # (NOT "individual-capstone-yourname") # Do NOT add a README, .gitignore, or license — keep it empty # 2. In your terminal, go to your existing capstone project cd ~/path/to/your/capstone-project # 3. Add your personal repo as a new remote git remote add personal https://github.com/YOUR-USERNAME/YOUR-REPO-NAME.git # 4. Push everything to your personal repo git push personal main # 5. Verify it worked — go to your new repo on GitHub and check
Pick a good repo name. Something like housing-price-predictor, health-risk-classifier, or movie-recommendation-app tells people what it does at a glance. Avoid generic names like capstone or ml-project.
Note: This keeps your original classroom repo intact. You're not deleting anything — just creating a copy on your personal profile.
2

Write a Portfolio-Ready README

Your README is the first thing anyone sees when they visit your repo. The template README from the assignment was great for building the project, but now you want something that sells what you built. Replace it with something like this:

# Your Project Title <!-- One line that explains what this does --> A machine learning web app that predicts [what it predicts] based on [key inputs]. <!-- If your app is hosted, put the link right here --> **[Try the live app](https://your-app-url.streamlit.app)** ![App Screenshot](docs/screenshot.png) ## The Problem <!-- 2-3 sentences: what question are you answering and why it matters --> ## What It Does - Takes user inputs for [key features] - Runs a trained [model type] model to predict [target] - Also classifies results into [categories] using [classification model] - Displays results with visualizations in an interactive Streamlit app ## Key Findings <!-- 3-4 bullet points about what you learned from the data --> ## Tech Stack - **Python** — Pandas, NumPy, Scikit-Learn - **Streamlit** — Interactive web application - **Models** — [Your regression model], [Your classification model] ## Run It Locally ```bash git clone https://github.com/YOUR-USERNAME/YOUR-REPO.git cd YOUR-REPO python -m venv venv source venv/bin/activate pip install -r requirements.txt streamlit run app/main.py ``` ## Project Structure ``` ├── app/ ← Streamlit application ├── data/ ← Dataset (raw and processed) ├── models/ ← Trained model files ├── notebooks/ ← EDA, feature engineering, modeling └── requirements.txt ```
Add a screenshot. Take a screenshot of your Streamlit app, save it as docs/screenshot.png, and reference it in the README. A visual makes a huge difference — people can see what your app looks like without even running it.

Host Your App So Anyone Can Use It

A live link in your README or on your resume is worth way more than "clone this repo and run it yourself." Here are your main options:

Heads up about free tier sleep behavior: On Streamlit Community Cloud (and most free tiers), your app goes to sleep if nobody uses it for a while. When someone visits after it's been sleeping, it takes 30–60 seconds to wake back up and load. This is totally normal and expected on free hosting — it's not broken, it just needs a moment to spin up. Your app will work fine once it's loaded.

My Recommendation: Streamlit Community Cloud

For most of you, this is the move. It's free, it's made for exactly what you built, and it takes about 2 minutes to deploy:

Quick Deploy Steps

  1. Go to share.streamlit.io and sign in with GitHub
  2. Click "New app"
  3. Select your repository and branch (main)
  4. Set the main file path to app/main.py (or wherever your Streamlit app lives)
  5. Click "Deploy" — that's it
Make sure your requirements.txt is accurate. Streamlit Cloud installs your dependencies from this file. If your app uses a package that's not listed, the deploy will fail. Also remove notebook and ipykernel from requirements if they're there — those are for local development only and can slow down or break the deploy.
4

Final Polish

A few more things that take 5 minutes each but make a real difference:

Add Custom Theming to Your App

I highly recommend this one. You can customize your Streamlit app's colors and fonts by adding a config file. Create a .streamlit/config.toml file in your project root:

# .streamlit/config.toml [theme] primaryColor = "#C65D00" # Buttons, sliders, links backgroundColor = "#FFFFFF" # Main background secondaryBackgroundColor = "#F5E6D3" # Sidebar background textColor = "#3D3D3D" # Main text font = "sans serif"

Change the colors to whatever fits your project. It doesn't need to be beautiful — just some intentional design choices make it stick out and not feel like default Streamlit. That small effort signals that you care about the end product.

Record a Demo Video

Free hosting can go down. A 30–60 second screen recording of your app in action is insurance. The best approach: upload it as an unlisted YouTube video and link it in your README. Unlisted means only people with the link can find it, but it's easy to share and never goes down. Walk through the app, enter some inputs, show the predictions.

Pin the Repo on Your GitHub Profile

Go to your GitHub profile, click "Customize your pins", and pin your capstone repo. This makes it the first thing anyone sees when they visit your GitHub. You get 6 pins — this project should be one of them.

Add It to LinkedIn

Two places to put it:

  • Projects section — Add the project with a description, the live app link, and the GitHub link
  • A post (optional) — Write a short post about what you built. Include the live link and a screenshot. This is definitely optional, but you may be surprised by how much engagement you get. People look at your post history — having a project post on there shows you're building real things.

  • README has been updated with project description, screenshot, and how to run it

  • Demo video recorded (backup for when the app is sleeping)
  • Added to LinkedIn profile and/or posted about it
  • Custom theme added via .streamlit/config.toml
